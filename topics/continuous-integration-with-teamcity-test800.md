<p>Continuous Integration is a software development practice in which developers commit code changes into a shared repository several times a day. Each commit is followed by an automated build to ensure that new changes integrate well into the existing code base and to detect problems early.</p>
<p>To learn more about continuous integration basics, refer to <a href="http://www.martinfowler.com/articles/continuousIntegration.html" rel="nofollow">Martin Fowlers article</a>.</p>
<h2><a id="user-content-what-is-teamcity" class="anchor" aria-hidden="true" href="#what-is-teamcity"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>What is TeamCity?</h2>
<p>JetBrains TeamCity is a user-friendly continuous integration (CI) server for developers and build engineers <a href="http://www.jetbrains.com/teamcity/buy/index.jsp" rel="nofollow">free of charge with the Professional Server License</a> and easy to set up!</p>
<h3><a id="user-content-what-can-you-do-with-teamcity" class="anchor" aria-hidden="true" href="#what-can-you-do-with-teamcity"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>What can you do with TeamCity?</h3>
<ul>
<li>Run parallel builds simultaneously on different platforms and environments</li>
<li>Optimize the code integration cycle and be sure you never get broken code in the repository</li>
<li>Review on-the-fly test results reporting with intelligent tests re-ordering</li>
<li>Run code coverage and duplicates finder for Java and .NET</li>
<li>Customize statistics on build duration, success rate, code quality, and custom metrics</li>
<li>and much more.</li>
</ul>
<p>To learn more about major TeamCity features, refer to the <a href="http://www.jetbrains.com/teamcity/features/index.html" rel="nofollow">official JetBrains site</a>.</p>
<p>The complete list of supported platforms and environments can be found <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/supported-platforms-and-environments.md">here</a>.</p>
<h2><a id="user-content-basic-teamcity-concepts" class="anchor" aria-hidden="true" href="#basic-teamcity-concepts"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Basic TeamCity concepts</h2>
<p>This section explains the main concepts. The complete list can be found <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/supported-platforms-and-environments.md">here</a>.</p>
<p>The <strong>TeamCity build system</strong> comprises a server and <strong>Build Agents</strong>.</p>
<table><tbody><tr>
<td>
<p>Concept</p>
</td>
<td>
<p>Description</p>
</td></tr><tr>
<td>
<p><strong>Build Agent</strong></p>
</td>
<td>
<p>A piece of software that actually executes a build process. It is installed and configured separately from the TeamCity server, i.e. the agent can be installed on a separate machine (physical or virtual, and it can run the same operating system (OS) as the server or a different OS.
Build Agents in TeamCity can have different platforms, operating systems, and pre-configured environments that you may want to test your software on. Different types of tests can be run under different platforms simultaneously so the developers get faster feedback and more reliable testing results.</p>

<p>It is possible for the server and an agent to coexist on the same computer, but for production purposes, we recommend installing them on different machines for a number of reasons, the server performance being the most important.
</p>
</td></tr><tr>
<td>
<p><strong>TeamCity Server</strong></p>
</td>
<td>
<p>The <strong>server</strong> itself <strong>does not run either builds or tests:</strong> the server's job is to monitor all the connected build agents, distribute <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/build-queue.md">queued builds</a> to the agents based on compatibility requirements, and report the results. All information on the build results (build history and all the build-associated data except for artifacts and build logs), VCS changes, agents, build queue, user accounts and user permissions, etc. are stored in a <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/setting-up-an-external-database.md">database</a>.</p>

<p>It is possible for the server and an agent to coexist on the same computer, but for production purposes, we recommend installing them on different machines for a number of reasons, the server performance being the most important.
</p>
</td></tr><tr>
<td>
<p><strong>Project</strong></p>
</td>
<td>
<p>A TeamCity Project corresponds to a software project or a specific version/release of a software project. A project is a collection of build configurations.</p>
</td></tr><tr>
<td>
<p><strong>Build Configuration</strong></p>
</td>
<td>
<p>A combination of settings defining a build procedure. The settings include <strong>VCS Roots</strong>, <strong>Build Steps</strong>, <strong>Build Triggers</strong> described below.</p>
</td></tr><tr>
<td>
<p><strong>VCS Root</strong></p>
</td>
<td>
<p>A collection of version control settings (paths to sources, username, password, <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/vcs-checkout-mode.md">checkout mode</a> and other settings) that defines how TeamCity communicates with a version control (SCM) system to monitor changes and get sources for a build.</p>
</td></tr><tr>
<td>
<p><strong>Build Step</strong></p>
</td>
<td>
<p>A task to be executed. Each build step is represented by a <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/build-runner.md">build runner</a> providing integration with a specific build tool (like Ant, Gradle, MSBuild, etc), a testing framework (e.g. NUnit), or a code analysis engine. Thus, in a single build you can have several steps and sequentially invoke test tools, code coverage, and, for instance, compile your project.</p>
</td></tr><tr>
<td>
<p><strong>Build Trigger</strong></p>
</td>
<td>
<p>A  rule which initiates a new build on certain events. For example, a <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/configuring-vcs-triggers.md">VCS trigger</a> will automatically start a new build each time TeamCity detects a <strong>change</strong> in the configured <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/vcs-root.md">VCS roots</a>.</p>
</td></tr><tr>
<td>
<p><strong>Change</strong></p>
</td>
<td>
<p>Any modification of the source code which you introduce. If a change has been committed to the version control system, but not yet included in a build, it is considered pending for a certain build configuration.</p>
</td></tr><tr>
<td>
<p><strong>Build</strong></p>
</td>
<td>
<p>Refers to both: the actual process of creating an application version and the version itself.  After the build process is triggered, it is put into the <strong><a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/build-queue.md">build queue</a></strong> and is started when there are agents available to run it. After the build is finished, the build agent sends <strong><a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/build-artifact.md">Build Artifacts</a></strong> to the server.</p>
</td></tr><tr>
<td>
<p><strong>Build Queue</strong></p>
</td>
<td>
<p>A list of builds that were <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/configuring-build-triggers.md">triggered</a> and are waiting to be started. TeamCity will distribute them to <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/agent-requirements.md">compatible</a> build agents as soon as the agents become idle. A queued build is assigned to an agent at the moment when it is started on the agent; no pre-assignment is made while the build is waiting in the build queue.</p>
</td></tr><tr>
<td>
<p><strong>Build Artifacts</strong></p>
</td>
<td>
<p>Files produced by a build, for example, installers, WAR files, reports, log files, etc, when they become available for download.</p>
</td></tr></tbody></table>
<h3><a id="user-content-basic-ci-workflow-in-teamcity" class="anchor" aria-hidden="true" href="#basic-ci-workflow-in-teamcity"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Basic CI Workflow in TeamCity</h3>
<p><a target="_blank" rel="noopener noreferrer" href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/cicd-flow-test800.png"><img src="/JetBrains/teamcity-documentation/raw/testing-diagrams-supernova/topics/cicd-flow-test800.png" width="800" alt="Diagram 800px" style="max-width:100%;"></a></p>
<p>To understand the data flow between the server and the agents, what is passed to the agents, how and when TeamCity gets the results, let's take a look at a simple build lifecycle.</p>
<ol>
<li>The TeamCity server detects a change in your VCS Root and stores it in the database.</li>
<li>The build trigger sees the change in the database and adds a build to the queue.</li>
<li>The server finds an idle compatible build agent and assigns the queued build to this agent.</li>
<li>The agent executes the Build Steps. While the build steps are being executed, the build agent reports the build progress to the TeamCity server sending all the log messages, test reports, code coverage results, etc. to the server on the fly, so you can monitor the build process in real time.</li>
<li>After finishing the build, the build agent sends <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/build-artifact.md">Build Artifacts</a> to the server.</li>
</ol>
<p>Now you can proceed with <a href="/JetBrains/teamcity-documentation/blob/testing-diagrams-supernova/topics/installation-quick-start.md">TeamCity installation</a>.</p>
